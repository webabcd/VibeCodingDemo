# 个人规则（全局生效）
## 一、基础交互规则
1. 所有回复**必须使用中文**，代码注释优先中文，关键逻辑补充英文注释。
2. 回答**结论先行**，先给方案/代码，再解释细节，避免冗余。
3. 代码输出**完整可运行**，附带最小依赖与运行示例，禁止只给片段。
4. 遇到问题时，**优先查官方文档**（Python.org、PyPI、框架官网），不编造方案。

## 二、Python 编码规范（强制遵循）
### 1. 风格与格式
- 严格遵循 **PEP 8** 规范：4 空格缩进、行宽 ≤ 88、命名清晰{insert\_element\_0\_}。
- 变量/函数：`snake_case`；类：`PascalCase`；常量：`UPPER_SNAKE_CASE`{insert\_element\_1\_}。
- 导入顺序：标准库 → 第三方库 → 本地模块，分组空行，不使用 `*` 导入。
- 代码格式化：默认用 `ruff format`，类型检查用 `basedpyright`。

### 2. 代码质量
- 所有函数/类**必须加文档字符串**（Google 风格），含功能、参数、返回值、异常。
- 避免深层嵌套（≤ 3 层），**提前返回**，减少缩进复杂度{insert\_element\_2\_}。
- 禁止魔法数字、硬编码路径，用常量/配置文件替代。
- 异常处理精准，不裸 `except`，捕获具体异常并说明原因。

### 3. 工程化要求
- 项目使用 **uv** 管理虚拟环境与依赖，虚拟环境目录 `.venv`。
- 依赖文件：`pyproject.toml`（优先）+ `requirements.txt`（兼容）。
- 测试用 `pytest`，关键功能覆盖率 ≥ 80%，测试文件命名 `test_*.py`。
- 日志用 `logging` 模块，不使用 `print` 调试；配置分环境（dev/prod）。

## 三、代码生成规则
1. 生成代码时，**自动补全类型注解**（Python 3.10+），支持静态检查。
2. 工具/脚本优先写 **CLI 入口**（`argparse`/`click`），方便直接运行。
3. 涉及并发：优先 `asyncio`/`threading`，必须说明线程安全与资源释放。
4. 数据库操作：用 ORM（SQLAlchemy/Peewee），禁止原生 SQL 拼接防注入。

## 四、调试与优化规则
1. 报错时：**先给错误定位**，再给修复代码，最后解释原因与预防方法。
2. 性能优化：先分析瓶颈（`cProfile`/`line_profiler`），再给优化方案。
3. 重构：小步修改，每次改一个逻辑，**重构后必须跑通测试**。
4. 安全：敏感信息（密钥/密码）用环境变量/配置文件，不硬编码。

## 五、输出格式规则
- 代码块标注语言：```python ... ```，关键行加注释。
- 命令行示例：```bash ... ```，区分 Windows/macOS 命令。
- 配置/依赖：用表格或代码块，清晰易复制。
- 超过 30 行的代码，拆分函数/模块，不堆砌长文件。

## 六、禁用行为（严格禁止）
- 禁止生成无注释、无类型、无测试的“三无代码”。
- 禁止使用已弃用 API、不安全写法（如 `eval`/`exec` 非必要场景）。
- 禁止忽略依赖版本、环境差异，必须说明兼容范围。
- 禁止编造解决方案，不确定时明确说明并给出查证方向。
